Component Details (1/2)

1. Application Load Balancer

  • Type: Layer 7 (HTTP/HTTPS)
  • Scheme: Internet-facing
  • Subnets: 2 public (different AZs)
  • Security: HTTP (80), HTTPS (443)
  • Health Checks: /health.html every 30s

2. Target Group

  • Protocol: HTTP:80
  • Health Path: /health.html
  • Interval: 30 seconds
  • Healthy: 2 consecutive successes
  • Unhealthy: 3 consecutive failures

3. Auto Scaling Group

  • Template: asg-demo-template
  • Instance: t3.micro (2 vCPU, 1 GiB)
  • AMI: Amazon Linux 2
  • Min/Desired/Max: 1/2/5 instances
  • Health Check: ELB, 300s grace
  • AZs: us-east-1a, us-east-1b

Component Details (2/2)

4. EC2 Instances

  • OS: Amazon Linux 2
  • Web Server: Apache (httpd)
  • Stress Tool: stress (CPU testing)
  • Monitoring: CloudWatch Agent
  • User Data: Automated setup
  • IAM Role: CloudWatch permissions

5. Launch Template

  • Version: Latest
  • Instance Type: t3.micro
  • Security Group: Instance SG
  • Key Pair: asg-demo-key
  • IAM Profile: asg-demo-ec2-profile

CloudWatch Monitoring

1. CPU Utilization (0-100%)

  • Shows how hard your server is working
  • Example: 20% = relaxed, 80% = very busy
  • High CPU means you need more servers

2. Network In/Out (bytes)

  • Network In: Data coming TO your server
  • Network Out: Data leaving your server
  • Example: 1000 requests/min = high traffic

3. Disk Read/Write (bytes)

  • How much data your server reads/writes to storage
  • Example: Reading logs, writing uploads
  • High disk activity can slow down your app

4. Request Count (number)

  • Total HTTP requests to load balancer
  • Example: 100/min = normal, 10,000 = spike
  • Helps understand user traffic patterns

5. Target Response Time (ms)

  • How long your server takes to respond
  • Example: 50ms = fast, 2000ms = slow
  • Slow responses mean scale up needed

6. Healthy/Unhealthy Hosts

  • Healthy: Servers passing health checks ✅
  • Unhealthy: Servers failing checks ❌
  • Example: 2 healthy, 0 unhealthy = all good

ECS Components Explained

Container Layer

  • ECR: Stores Docker images
  • Task Definition: Blueprint for containers
  • Container: Running application instance

Orchestration Layer

  • ECS Cluster: Logical grouping
  • ECS Service: Maintains desired tasks
  • Fargate: Serverless compute engine

Network Layer

  • VPC: Private network
  • Subnets: Network segments
  • Security Groups: Firewall rules

Load Balancing

  • ALB: Distributes traffic
  • Target Group: Container endpoints
  • Health Checks: Container monitoring
